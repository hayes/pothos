Pothos is a library for building GraphQL schemas in TypeScript. You define your types, fields, and resolvers with a builder, and the types flow from your data through that builder into every resolver — `parent`, `args`, and `ctx` arrive typed because the builder's generics carry them, not because anything scans your code. You describe the schema once, in TypeScript: no SDL to keep in sync, no decorators, no code-generation step.

`builder.toSchema()` hands you a standard graphql-js `GraphQLSchema` at startup, so any server — [GraphQL Yoga](https://the-guild.dev/graphql/yoga-server), Apollo — can run it, and your resolvers stay ordinary functions. The heavy lifting happens in the type system and at schema-build time; at request time there's nothing between the client and your resolver. Core depends only on `graphql`. Everything past it — Relay, scope-auth, errors, validation, dataloaders, Prisma, Drizzle, federation — is a plugin that extends the same builder.

## How it fits together

The example above is the whole pipeline in miniature:

- **Your data leads.** The backing model — `{ id: string; name: string }` — is the shape your data already has. `builder.objectRef<T>()` types a GraphQL object by that model, so the schema follows your data instead of the other way around.
- **Resolvers are typed from it.** `frodo` has to return that shape; return the wrong thing and the file doesn't compile. There's no second type definition to keep in step.
- **The output is a plain schema.** `builder.toSchema()` produces an ordinary graphql-js `GraphQLSchema`. From there it's just GraphQL — hand it to any server and run it.

Add a plugin and the same builder grows new methods — `t.connection`, `authScopes` on any field, `t.prismaField` — that sit next to the ones you already use and share the same inference. That's the entire model: data → builder → typed resolvers → a standard schema → your server. It runs at scale, too; Pothos is in production at companies including Airbnb and Netflix.
