import { Callout } from 'fumadocs-ui/components/callout';

The Drizzle plugin builds GraphQL object types directly from your Drizzle tables. You declare a type with `builder.drizzleObject`, expose columns, and add relations with `t.relation` — the plugin reads your Drizzle relational schema and translates each GraphQL selection into a single, precisely-scoped query. Nothing is over-fetched: a field is only loaded when the client asks for it.

The plugin is built on Drizzle's [relational query builder](https://rqbv2.drizzle-orm-fe.pages.dev/docs/relations-v2), so it requires that you define and configure all the relevant relations in your Drizzle schema first.

<Callout type="warn">
  This package is new and depends on Drizzle's [RQB v2 API](https://rqbv2.drizzle-orm-fe.pages.dev/docs/rqb-v2#include-custom-fields). Some features are still missing and the API may change. It currently requires the `beta` tag for `drizzle-orm`.

  Upgrading from an older version of this plugin? Read the [Drizzle relations migration guide](https://rqbv2.drizzle-orm-fe.pages.dev/docs/relations-v1-v2) and check this package's changelog for Pothos-specific changes.
</Callout>

## Install

```package-install
npm install --save @pothos/plugin-drizzle drizzle-orm@beta
```

Once your Drizzle schema and relations are configured, register the plugin on the builder. Pass the Drizzle `client`, the dialect-specific `getTableConfig`, and your `relations`, and expose the relations type through `DrizzleRelations` so Pothos can infer table shapes:

```ts
import { drizzle } from 'drizzle-orm/...';
// Import the getTableConfig for your dialect
import { getTableConfig } from 'drizzle-orm/sqlite-core';
import SchemaBuilder from '@pothos/core';
import DrizzlePlugin from '@pothos/plugin-drizzle';
import { relations } from './db/relations';

const db = drizzle({ client, relations });

export interface PothosTypes {
  DrizzleRelations: typeof relations;
}

const builder = new SchemaBuilder<PothosTypes>({
  plugins: [DrizzlePlugin],
  drizzle: {
    client: db, // or (ctx) => db to build a request-specific client
    getTableConfig,
    relations,
  },
});
```

Pass `client` as a function of `ctx` when each request needs its own client — for per-request transactions or connection scoping.

### Working with Relay and With-Input

The Drizzle plugin integrates with several other plugins. [Relay](./relay) and [with-input](./with-input) aren't required, but the Relay integration (nodes and connections) and many examples below assume they're installed:

```ts
import { drizzle } from 'drizzle-orm/...';
import SchemaBuilder from '@pothos/core';
import DrizzlePlugin from '@pothos/plugin-drizzle';
import RelayPlugin from '@pothos/plugin-relay';
import WithInputPlugin from '@pothos/plugin-with-input';
import { getTableConfig } from 'drizzle-orm/sqlite-core';
import { relations } from './db/relations';

const db = drizzle({ client, relations });

export interface PothosTypes {
  DrizzleRelations: typeof relations;
}

const builder = new SchemaBuilder<PothosTypes>({
  plugins: [RelayPlugin, WithInputPlugin, DrizzlePlugin],
  drizzle: {
    client: db,
    getTableConfig,
    relations,
  },
});
```

## Defining objects

`builder.drizzleObject` defines a GraphQL object type from a Drizzle table. The first argument is the table name from your relations; `name` sets the GraphQL type name:

```ts
const PlayerRef = builder.drizzleObject('players', {
  name: 'Player',
  fields: (t) => ({
    firstName: t.exposeString('firstName'),
    lastName: t.exposeString('lastName'),
  }),
});
```

You can expose any column, and GraphQL field names don't need to match column names. `drizzleObject` returns an `ObjectRef` — `PlayerRef` here — that behaves like any other [object ref](../fundamentals/objects): use it as a field `type`, pass it to `t.variant`, or hand it to `t.drizzleField`.

## Custom fields

Fields often don't map to a single column. Add a resolver like you would on any Pothos object type — the row is the `parent`:

```ts
const PlayerRef = builder.drizzleObject('players', {
  name: 'Player',
  fields: (t) => ({
    fullName: t.string({
      resolve: (player) => `${player.firstName} ${player.lastName}`,
    }),
  }),
});
```

## Selecting columns

By default every column of the table is available on `parent`. For wide tables it's more efficient to select only what you need. A `select` option on the type controls the default selection — `columns`, related tables via `with`, and computed SQL via `extras`:

```ts
const PlayerRef = builder.drizzleObject('players', {
  name: 'Player',
  select: {
    columns: {
      firstName: true,
      lastName: true,
    },
    with: {
      profile: true,
    },
    extras: {
      lowercaseName: (players, sql) => sql<string>`lower(${players.firstName})`,
    },
  },
  fields: (t) => ({
    fullName: t.string({
      resolve: (player) => `${player.firstName} ${player.lastName}`,
    }),
    bio: t.string({
      resolve: (player) => player.profile.bio,
    }),
    email: t.string({
      resolve: (player) => `${player.lowercaseName}@ultimate.example`,
    }),
  }),
});
```

Anything selected on the type is available in every resolver on that type. Columns you don't select can still be exposed with `t.exposeString` and friends — the plugin adds them to the query when the field is requested.

### Per-field selections

A type-level `select` loads its columns for every request. To load a column only when a specific field is queried, put `select` on the field instead. This keeps each field's cost tied to whether the client asks for it:

```ts
const PlayerRef = builder.drizzleObject('players', {
  name: 'Player',
  select: {},
  fields: (t) => ({
    fullName: t.string({
      select: {
        columns: { firstName: true, lastName: true },
      },
      resolve: (player) => `${player.firstName} ${player.lastName}`,
    }),
    bio: t.string({
      select: {
        with: { profile: true },
      },
      resolve: (player) => player.profile.bio,
    }),
    email: t.string({
      select: {
        extras: {
          lowercaseName: (players, sql) => sql<string>`lower(${players.firstName})`,
        },
      },
      resolve: (player) => `${player.lowercaseName}@ultimate.example`,
    }),
  }),
});
```

## Relations

`t.relation` turns a Drizzle relation into a GraphQL field. The plugin reads the relation from your schema and gives the field the correct type automatically:

```ts
builder.drizzleObject('profiles', {
  name: 'Profile',
  fields: (t) => ({
    bio: t.exposeString('bio'),
  }),
});

builder.drizzleObject('reports', {
  name: 'Report',
  fields: (t) => ({
    title: t.exposeString('title'),
    author: t.relation('author'),
  }),
});

builder.drizzleObject('players', {
  name: 'Player',
  fields: (t) => ({
    firstName: t.exposeString('firstName'),
    profile: t.relation('profile'),
    reports: t.relation('reports'),
  }),
});
```

### Filtering and ordering relations

Pass a `query` option to filter, order, or paginate a relation — and add `args` to make it client-driven. `query` can be a static object or a callback that receives `(args, ctx, pathInfo)`, where `pathInfo` carries the GraphQL query path (`path` and `segments`):

```ts
builder.drizzleObject('players', {
  name: 'Player',
  fields: (t) => ({
    firstName: t.exposeString('firstName'),
    reports: t.relation('reports', {
      args: {
        limit: t.arg.int(),
        offset: t.arg.int(),
      },
      query: (args) => ({
        limit: args.limit ?? 10,
        offset: args.offset ?? 0,
        where: {
          published: true,
        },
        orderBy: {
          updatedAt: 'desc',
        },
      }),
    }),
    // The same relation, filtered a second way
    drafts: t.relation('reports', {
      query: {
        where: {
          published: false,
        },
      },
    }),
  }),
});
```

The `query` options map straight onto the relational query builder — see the [relation query docs](https://orm.drizzle.team/docs/rqb#querying) for the full set.

## Entry points with drizzleField

Objects and relations describe the graph, but the client needs a way in. `t.drizzleField` defines a field that returns a Drizzle object — on `Query`, or on any other object type. Its resolver receives a `query` function that you **must** call and pass to a Drizzle `findFirst` or `findMany`. `query` merges the selection needed for the nested GraphQL fields with any options you pass it:

```ts
builder.queryType({
  fields: (t) => ({
    report: t.drizzleField({
      type: 'reports',
      args: {
        id: t.arg.id({ required: true }),
      },
      resolve: (query, root, args, ctx) =>
        db.query.reports.findFirst(
          query({
            where: {
              id: Number.parseInt(args.id, 10),
            },
          }),
        ),
    }),
    reports: t.drizzleField({
      type: ['reports'],
      resolve: (query, root, args, ctx) => db.query.reports.findMany(query()),
    }),
  }),
});
```

Use a single table name (`type: 'reports'`) for one record, or a list (`type: ['reports']`) for many.

## Variants

Sometimes one table backs more than one GraphQL type — a public `Player` and a private `Viewer` view of the same row, say. That's what `variants` are for. A variant has three parts:

- Pass `variant` instead of `name` to `drizzleObject` (or `drizzleNode`).
- Pass an `ObjectRef` to the `type` option of `t.relation` and similar fields.
- Use `t.variant` to add a field that references another variant of the same record.

```ts
// A private view of the current player
export const Viewer = builder.drizzleObject('players', {
  variant: 'Viewer',
  select: {},
  fields: (t) => ({
    id: t.exposeID('id'),
    // A reference to the public Player type, so its fields can be queried here too
    player: t.variant('players'),
    // Exposing drafts on Viewer lets a player see their own unpublished
    // reports without exposing them for other players in the API
    drafts: t.relation('reports', {
      query: {
        where: {
          published: false,
        },
        orderBy: {
          updatedAt: 'desc',
        },
      },
    }),
  }),
});

builder.queryType({
  fields: (t) => ({
    me: t.drizzleField({
      // The ref returned by drizzleObject can be the drizzleField type
      type: Viewer,
      resolve: (query, root, args, ctx) =>
        db.query.players.findFirst(
          query({
            where: {
              id: ctx.user.id,
            },
          }),
        ),
    }),
  }),
});

builder.drizzleNode('players', {
  name: 'Player',
  id: { column: (player) => player.id },
  fields: (t) => ({
    firstName: t.exposeString('firstName'),
    // Resolves to Viewer, or null when this row isn't the current player
    viewer: t.variant(Viewer, {
      isNull: (player, args, ctx) => player.id !== ctx.user?.id,
    }),
  }),
});
```

## Derived fields with relatedField

`t.relatedField` defines a field from a relation using a custom selection — useful for exposing derived data, like an aggregate, without loading the full related records. The `select` callback receives a `buildFilter` helper that produces the correct `WHERE` clause for the relation:

```ts
builder.drizzleNode('players', {
  name: 'Player',
  id: { column: (player) => player.id },
  fields: (t) => ({
    firstName: t.exposeString('firstName'),
    // A count of related reports, computed in SQL
    reportsCount: t.relatedField('reports', {
      type: 'Int',
      select: (buildFilter) => ({
        extras: {
          reportsCount: (parent) => db.$count(reports, buildFilter(parent)),
        },
      }),
      resolve: (player) => player.reportsCount,
    }),
  }),
});
```

This is `t.field` under the hood; `buildFilter` just makes it easy to scope the query to the related records.

### Related counts

Counting related records is common enough to have a dedicated `t.relatedCount` that handles the boilerplate:

```ts
builder.drizzleNode('players', {
  name: 'Player',
  id: { column: (player) => player.id },
  fields: (t) => ({
    firstName: t.exposeString('firstName'),
    // Count of all related comments
    commentsCount: t.relatedCount('comments'),
    // Count with a static filter
    publishedReportsCount: t.relatedCount('reports', {
      where: eq(reports.published, true),
    }),
  }),
});
```

`where` accepts a static SQL filter or a function of the field's args and context:

```ts
publishedReportsCount: t.relatedCount('reports', {
  args: {
    category: t.arg.string(),
  },
  where: (args, ctx) =>
    args.category
      ? and(eq(reports.published, true), eq(reports.category, args.category))
      : eq(reports.published, true),
});
```

## Relay nodes and connections

With the [Relay plugin](./relay) installed, the Drizzle plugin can define Relay `nodes` and `connections` for you, so complying with the Relay spec takes almost no extra code.

### Nodes

`builder.drizzleNode` works like `drizzleObject` but requires an `id` — the column to use for the node's global id:

```ts
builder.drizzleNode('players', {
  name: 'Player',
  id: {
    column: (player) => player.id,
    // other options for the ID field can be passed here
  },
  fields: (t) => ({
    firstName: t.exposeString('firstName'),
    lastName: t.exposeString('lastName'),
  }),
});
```

For a composite primary key, set `column` to a list of columns.

### Related connections

Use `t.relatedConnection` instead of `t.relation` to expose a relation as a connection. It defines the `Connection` and `Edge` types and their fields automatically. Extra arguments customize the connection and edge types, exactly like `t.connection` from the [Relay plugin](./relay):

```ts
builder.drizzleNode('players', {
  name: 'Player',
  id: { column: (player) => player.id },
  fields: (t) => ({
    reports: t.relatedConnection('reports'),
  }),
});
```

You can pass a `query` just like on `t.relation`, with one difference in the `orderBy` format. To support backwards pagination efficiently, some queries run in reverse, which means inverting the `orderBy`. So `t.relatedConnection` takes `orderBy` as an object — `{ asc: column }` or `{ desc: column }` — rather than Drizzle's `asc(column)` / `desc(column)` helpers. It can still be a single column or an array for multi-column ordering. Ordering defaults to the table's primary key, and the `orderBy` columns are also used to derive the cursor:

```ts
builder.drizzleNode('players', {
  name: 'Player',
  id: { column: (player) => player.id },
  fields: (t) => ({
    reports: t.relatedConnection('reports', {
      query: () => ({
        where: {
          published: true,
        },
        orderBy: {
          id: 'desc',
        },
      }),
    }),
  }),
});
```

Set `totalCount: true` to add a `totalCount` field to the connection:

```ts
builder.drizzleNode('players', {
  name: 'Player',
  id: { column: (player) => player.id },
  fields: (t) => ({
    reports: t.relatedConnection('reports', {
      totalCount: true,
      query: () => ({
        where: {
          published: true,
        },
        orderBy: {
          id: 'desc',
        },
      }),
    }),
  }),
});
```

The count runs only when `totalCount` is actually requested, and it's issued as a subquery within the main query for efficiency:

```graphql
query {
  player(id: "...") {
    reports(first: 10) {
      totalCount
      edges {
        node {
          id
          title
        }
      }
    }
  }
}
```

### Connections as entry points

Like `t.drizzleField`, `t.drizzleConnection` defines a connection field that's an entry point into your Drizzle data. Its `orderBy` behaves the same as `t.relatedConnection`:

```ts
builder.queryFields((t) => ({
  reports: t.drizzleConnection({
    type: 'reports',
    resolve: (query, root, args, ctx) =>
      db.query.reports.findMany(
        query({
          where: {
            published: true,
          },
          orderBy: {
            id: 'desc',
          },
        }),
      ),
  }),
}));
```

Add a `totalCount` callback to include a total count. It receives the standard resolver arguments (`parent`, `args`, `context`, `info`), so the count can depend on the query context — the example uses `db.$count()`, but any Drizzle count works:

```ts
builder.queryFields((t) => ({
  reports: t.drizzleConnection({
    type: 'reports',
    totalCount: () => db.$count(reports, eq(reports.published, true)),
    resolve: (query, root, args, ctx) =>
      db.query.reports.findMany(
        query({
          where: {
            published: true,
          },
          orderBy: {
            id: 'desc',
          },
        }),
      ),
  }),
}));
```

When only `totalCount` is requested — without `edges` or `nodes` — the main query is skipped and only the count runs.

## Connection helpers for indirect relations

`t.relatedConnection` covers most many-to-many connections defined through Drizzle relations. When a connection doesn't map to a single relation — paginating through a join table, or paginating a relation whose nodes live a level deeper — reach for `drizzleConnectionHelpers`, which lets you build the connection with the plain `t.connection` API.

The example below paginates a `memberships` join table, but resolves each node to the `team` nested one level deeper:

```typescript
// The node type for the connection
const Team = builder.drizzleObject('teams', {
  name: 'Team',
  fields: (t) => ({
    id: t.exposeID('id'),
    name: t.exposeString('name'),
  }),
});

// Helpers that know how to select and resolve nodes for this connection
const membershipsConnection = drizzleConnectionHelpers(builder, 'memberships', {
  // select the data needed for the nodes
  select: (nestedSelection) => ({
    with: {
      // nestedSelection builds the correct selection for the node
      team: nestedSelection(),
    },
  }),
  // resolve the node from each returned list item
  resolveNode: (membership) => membership.team,
});

builder.drizzleObjectField('Player', 'teamsConnection', (t) =>
  t.connection({
    // The Node type
    type: Team,
    // Since this isn't t.relatedConnection, include the selection manually
    select: (args, ctx, nestedSelection) => ({
      with: {
        memberships: membershipsConnection.getQuery(args, ctx, nestedSelection),
      },
    }),
    // Format the loaded rows for the connection
    resolve: (player, args, ctx) =>
      membershipsConnection.resolve(player.memberships, args, ctx, player),
  }),
);
```

Pagination args apply to the relation to the join table (`memberships`); the nodes themselves are the `team` nested inside each membership.

`drizzleConnectionHelpers` also handles the case where the edge and node are the same model, and pagination happens directly on a relation to the node type (even a nested one). With no options, the helper's `ref` is the node type:

```ts
const commentConnectionHelpers = drizzleConnectionHelpers(builder, 'comments');

const SelectReport = builder.drizzleObject('reports', {
  fields: (t) => ({
    title: t.exposeString('title'),
    comments: t.connection({
      type: commentConnectionHelpers.ref,
      select: (args, ctx, nestedSelection) => ({
        with: {
          comments: commentConnectionHelpers.getQuery(args, ctx, nestedSelection),
        },
      }),
      resolve: (parent, args, ctx) =>
        commentConnectionHelpers.resolve(parent.comments, args, ctx),
    }),
  }),
});
```

### Arguments, ordering, and filtering

Define additional args, a default order, and a filter directly in the helper. Add the helper's args to the field with `getArgs`:

```ts
const membershipsConnection = drizzleConnectionHelpers(builder, 'memberships', {
  // define additional arguments
  args: (t) => ({}),
  query: (args) => ({
    // default order
    orderBy: {
      teamId: 'asc',
    },
    // default filter
    where: {
      accepted: true,
    },
  }),
  select: (nestedSelection) => ({
    with: {
      team: nestedSelection(),
    },
  }),
  resolveNode: (membership) => membership.team,
});

builder.drizzleObjectField('Player', 'teamsConnection', (t) =>
  t.connection({
    type: Team,
    // pull the args from the helper onto the field
    args: membershipsConnection.getArgs(),
    select: (args, ctx, nestedSelection) => ({
      with: {
        memberships: membershipsConnection.getQuery(args, ctx, nestedSelection),
      },
    }),
    resolve: (player, args, ctx) =>
      membershipsConnection.resolve(player.memberships, args, ctx, player),
  }),
);
```

### Fields on the edge

To expose data from the join row on the edge itself, use `drizzleObjectFields` (or `drizzleObjectField`) and pass edge options as the third argument to `t.connection`. The edge's parent shape is inferred from the connection's `resolve`:

```typescript
const membershipsConnection = drizzleConnectionHelpers(builder, 'memberships', {
  select: (nestedSelection) => ({
    with: {
      team: nestedSelection(),
    },
  }),
  resolveNode: (membership) => membership.team,
});

builder.drizzleObjectFields('Player', (t) => ({
  teamsConnection: t.connection(
    {
      type: Team,
      select: (args, ctx, nestedSelection) => ({
        with: {
          memberships: membershipsConnection.getQuery(args, ctx, nestedSelection),
        },
      }),
      resolve: (player, args, ctx) =>
        membershipsConnection.resolve(player.memberships, args, ctx, player),
    },
    {},
    // options for the edge object
    {
      fields: (edge) => ({
        createdAt: edge.field({
          type: 'DateTime',
          resolve: (membership) => membership.createdAt,
        }),
      }),
    },
  ),
}));
```

### Non-relation connections

`drizzleConnectionHelpers` also builds connections where the edges and nodes come from different types and there's no direct relation to lean on. Merge the `where` clause the helper generates with any additional filter you apply, so the two don't clobber each other:

```typescript
const membershipsConnection = drizzleConnectionHelpers(builder, 'memberships', {
  select: (nestedSelection) => ({
    with: {
      team: nestedSelection(),
    },
  }),
  resolveNode: (membership) => membership.team,
});

builder.queryFields((t) => ({
  teams: t.connection({
    type: Team,
    args: {
      playerId: t.arg.int({ required: true }),
    },
    nodeNullable: true,
    resolve: async (_, args, ctx, info) => {
      const query = membershipsConnection.getQuery(args, ctx, info);
      const memberships = await db.query.memberships.findMany({
        ...query,
        where: {
          ...query.where,
          playerId: args.playerId,
        },
      });
      return membershipsConnection.resolve(memberships, args, ctx);
    },
  }),
}));
```
