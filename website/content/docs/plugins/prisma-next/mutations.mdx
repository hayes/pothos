---
title: Mutations
description: t.prismaField for create, update, delete, upsert
---

There's no `t.prismaMutation` — `t.prismaField` covers mutations
directly. The same `apply` callable runs at the resolve site whether
the field is on `Query` or `Mutation`, so create / update / delete /
upsert flow through the same shape as a query.

## Unified resolve pattern

```ts
builder.mutationType({
  fields: (t) => ({
    createUser: t.prismaField({
      type: 'User',
      args: {
        email: t.arg.string({ required: true }),
        firstName: t.arg.string({ required: true }),
      },
      resolve: (apply, _parent, args, ctx) =>
        apply(ctx.db.orm.User).create({
          email: args.email,
          firstName: args.firstName,
        }),
    }),
  }),
});
```

`apply(ctx.db.orm.User)` returns the User collection with the
auto-include chain applied. The orm-client's `.create()` honors the
chain's `.include(...)` at runtime, so the inserted row comes back with
the requested relations preloaded — one round-trip, no follow-up read.

## Update, delete, upsert

```ts
updateUserName: t.prismaField({
  type: 'User',
  args: {
    id: t.arg.string({ required: true }),
    firstName: t.arg.string({ required: true }),
  },
  resolve: (apply, _p, args, ctx) =>
    apply(ctx.db.orm.User.where((u) => u.id.eq(args.id)))
      .update({ firstName: args.firstName }),
}),

deleteUser: t.prismaField({
  type: 'User',
  args: { id: t.arg.string({ required: true }) },
  resolve: (apply, _p, args, ctx) =>
    apply(ctx.db.orm.User.where((u) => u.id.eq(args.id))).delete(),
}),

upsertUser: t.prismaField({
  type: 'User',
  args: {
    email: t.arg.string({ required: true }),
    firstName: t.arg.string({ required: true }),
  },
  resolve: (apply, _p, args, ctx) =>
    apply(ctx.db.orm.User).upsert({
      where: { email: args.email },
      create: { email: args.email, firstName: args.firstName },
      update: { firstName: args.firstName },
    }),
}),
```

Chain `.where(...)` on the orm handle BEFORE `apply(...)` for update /
delete — the predicate scopes the mutation; `apply` layers includes on
top so the returned rows come back with the GraphQL selection set
preloaded.

## Single round-trip with includes

A `createUser` whose GraphQL query asks for `user.posts` inserts the
user AND fetches its posts in one round-trip:

```graphql
mutation {
  createUser(email: "a@b.com", firstName: "A") {
    id
    posts { id title }
  }
}
```

The orm-client's `.create(...)` sees the include chain layered onto the
collection and emits the matching `RETURNING` / follow-up select
inline.

## Composite mutations

For mutations that operate on filtered sets:

```ts
import { and } from '@pothos/plugin-prisma-next';

publishAllDrafts: t.prismaField({
  type: ['Post'],
  args: { authorId: t.arg.string({ required: true }) },
  resolve: (apply, _p, args, ctx) =>
    apply(
      ctx.db.orm.Post.where((p) =>
        and(p.authorId.eq(args.authorId), p.published.eq(0)),
      ),
    ).update({ published: 1 }),
}),
```

The mapper's includes still hoist into the returned set.
