---
title: Fields
description: Add fields to an object type and control what each one returns.
---

A type's fields are defined by the `fields` function you pass when you implement it. That function receives a field builder — conventionally named `t` — whose methods each return one field definition. The surrounding object definition (the ref, its backing model, and `implement`) is covered in [Object types](./objects).

## Exposing properties

When a field returns a property of the backing object unchanged, `t.expose*` forwards it for you, without a resolver:

<include cwd lang="typescript" meta='playground example="fundamentals-fields"'>playground-examples/fundamentals-fields/schema.ts#expose</include>

`t.exposeID`, `t.exposeString`, `t.exposeInt`, `t.exposeFloat`, and `t.exposeBoolean` each take the name of a property to read and produce a field of the matching scalar type; `id`, `name`, and `biography` here read straight from the `Character` object. Each also has a `…List` form (`t.exposeStringList` and so on) for properties that hold arrays. The field itself is named by the key you assign it to, so `bio: t.exposeString('biography')` would expose the same property under a shorter name.

## Computed fields

When a field's value isn't a property already sitting on the backing object, you give it a resolver. The scalar builders — `t.string`, `t.int`, `t.float`, `t.boolean`, and `t.id` — take a `resolve` function whose first argument is the backing object:

<include cwd lang="typescript">playground-examples/fundamentals-fields/schema.ts#computed</include>

`age` computes an `Int` from the character's `birthYear`. `race` returns another object type rather than a scalar, so it uses `t.field`, which takes an explicit `type`; the scalar builders are shorthands for `t.field` with the type already set. The resolver's first argument is always the backing object — the [Resolvers](./resolvers) guide covers the rest of its signature and what you can return.

## Lists

A field returns a list when its type is wrapped in an array. For a property that already holds an array, the `…List` expose helpers forward it:

<include cwd lang="typescript">playground-examples/fundamentals-fields/schema.ts#list</include>

`titles` exposes the `string[]` property as a `[String]` field. To build a list in a resolver, wrap the type the same way: `t.field({ type: [Faction] })` returns a list of `Faction`, and the resolver returns an array of the matching shape.

## Nullability

Every field defined so far is nullable. Pothos output fields are nullable by default, so `t.exposeID('id')` produces `id: ID` and any resolver may return `null`. Pass `nullable: false` to require a value:

```typescript
id: t.exposeID('id', { nullable: false }),
```

That makes the field `id: ID!`. A list field has two positions that can be null — the list itself and each of its items — so `nullable` also accepts a `{ list, items }` object to set them independently:

```typescript
titles: t.exposeStringList('titles', { nullable: { list: false, items: true } }),
```

The default can also be flipped for the whole schema, so fields are non-nullable unless marked; [Default nullability](../patterns/default-nullability) covers how.

## Splitting fields across files

A type doesn't have to define all of its fields in one place. `builder.objectField` adds a single field to a type that's already been referenced, from anywhere in your codebase:

<include cwd lang="typescript">playground-examples/fundamentals-fields/schema.ts#split</include>

This defines `Character.factions` in whatever module owns factions instead of in the `Character` module. `builder.objectFields` adds several fields at once. These calls can run before or after the type is implemented; Pothos collects the definitions and merges them when the schema builds, and defining the same field name twice is a build-time error. [Project layout](../patterns/project-layout) covers organizing a schema this way.
