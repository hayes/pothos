A union returns one of several object types that don't share fields. Where an [interface](./interfaces) demands every implementer carry common fields like `id` and `name`, a union groups types purely by the question they answer: "what is this thing?"

<includeregions cwd lang="typescript" meta='playground example="fundamentals-unions"'>playground-examples/fundamentals-unions/schema.ts#search-result-union,search-query</includeregions>

Each value the resolver returns has to be assignable to one of the `types` listed. `resolveType` picks which one.

## When to reach for a union

Unions fit a handful of recurring shapes:

- **Search results.** A query returns a heterogeneous mix — characters, locations, items — that share no fields beyond an id.
- **Activity feeds.** Each entry is one of `PostCreated`, `CommentAdded`, `UserJoined`, with different payloads.
- **Mutation outcomes.** A result that's either the changed entity or one of several error types — the pattern [`plugin-errors`](../plugins/errors) automates for you.

If the types do share fields, prefer an interface; clients can then select common fields without the per-type fragment dance.

## resolveType

Same rules as interfaces: return the concrete type's name as a string. With a `kind` discriminator on the backing model this is a one-liner:

```typescript
resolveType: (val) => val.kind,
```

Without a discriminator, infer from the shape:

```typescript
resolveType: (val) => {
  if ('terrain' in val) return 'Location';
  if ('text' in val) return 'Quote';
  return 'Character';
},
```

## Querying

Clients access every field through a per-type fragment, since the union itself has no fields:

```graphql
query Search {
  search(term: "shire") {
    __typename
    ... on Character {
      id
      name
    }
    ... on Location {
      name
      terrain
    }
    ... on Quote {
      text
    }
  }
}
```

`__typename` is always available on union fields — it's the discriminator clients use to render the right component.
