Most of the types in a GraphQL schema are object types. This page covers defining them, and the backing model that connects each type to the data your resolvers return.

## Defining an object type

An object type is defined in two steps: create a reference to it, then implement its fields. The same type can be written a few other ways, shown in the tabs below; the rest of this page uses `objectRef`.

<includeregions cwd lang="typescript" meta='playground example="fundamentals-objects-variant-classes" tab="Classes"'>playground-examples/fundamentals-objects/variant-classes/schema.ts#character-class,object-type</includeregions>

<includeregions cwd lang="typescript" meta='playground example="fundamentals-objects-variant-builder-types" tab="Builder types"'>playground-examples/fundamentals-objects/variant-builder-types/schema.ts#character-model,builder-init,object-type</includeregions>

`builder.objectRef` creates a reference to a new `Character` type. The generic parameter — `ICharacter` here — is the TypeScript shape of the data behind the type (Pothos calls this the backing model). `implement` then defines the type's fields: `t.exposeID('id')` and `t.exposeString('name')` return those properties straight from the backing object. The [Fields](./fields) guide covers the field builder in detail.

## The backing model

The backing model is the TypeScript type a resolver returns for an object type, and the type Pothos hands back as `parent` to every field defined on it. You state it once, as the generic on `objectRef`, and Pothos checks the rest of the type against it: `t.exposeString('name')` only compiles when `name` is a string on `ICharacter`, and any field that returns a `Character` must resolve to a value matching `ICharacter`. When your data source already produces that shape (a database row, an ORM model, an in-memory record), you usually return it untouched.

The GraphQL type does not have to expose everything in the backing model. `ICharacter` carries `editorId` and `biography`, but the type above exposes only `id` and `name`; the extra properties stay available to resolvers without appearing in the schema.

## Object type options

`objectRef` takes only the type's name. Every other option for the type is passed to `implement`: the type's `fields`, a `description`, the [interfaces](./interfaces) it implements, and an `isTypeOf` function for [resolving abstract types](./interfaces). `fields` is the one you pass most often, but it works like any other option on `implement`.

When a type doesn't need to be referenced before it's implemented, you can chain the two calls into one expression:

<includeregions cwd lang="typescript" meta='playground example="fundamentals-objects"'>playground-examples/fundamentals-objects/schema.ts#race</includeregions>

`builder.objectRef<IRace>('Race').implement({ ... })` creates the reference and implements it in the same statement. Keeping them apart — as `Character` does above — is what lets two types reference each other, since the reference exists before either type's fields are defined. The [Circular references](../patterns/circular-references) pattern covers that case.

## Returning an object type

A reference can be used as the `type` of any field, on any type in the schema. Here the root `Query` type returns lists of `Character` and `Race`:

<includeregions cwd lang="typescript" meta='playground example="fundamentals-objects"'>playground-examples/fundamentals-objects/schema.ts#query</includeregions>

`type: [Character]` makes the field a list of the `Character` type; a bare `type: Character` returns a single one. Because the field returns `Character`, its resolver has to return values matching `ICharacter`, and TypeScript reports an error if it doesn't. The [Queries](./queries) guide covers the `Query` root itself.

## Classes and builder types

The **Classes** and **Builder types** tabs above define the same `Character` type two other ways. All three are fully supported, and a schema can mix them freely.

**Classes.** `builder.objectType(Character, { ... })` takes a class and uses its instance type as the backing model, so `parent` in every resolver is an instance of the class, methods and getters included. This fits well when your app already keeps classes for its data. A class is also a value at runtime, so you can write `isTypeOf: (value) => value instanceof Character` — an `instanceof` check that tells Pothos which member a value is when it resolves an [interface or union](./interfaces).

**Builder types.** Registering a type on the `Objects` generic maps a type name to its backing model, which lets you refer to the type by its string name — `t.field({ type: 'Character' })` — anywhere in the schema. This keeps your type definitions in one place instead of importing a reference into every file that uses them.
