Mutations are queries that change state. The API mirrors [Queries](./queries) â€” `mutationType`, `mutationField`, `mutationFields` â€” but the patterns inside change. A typical mutation takes an input object, checks the request context, performs the change, and returns the updated entity.

```typescript playground example="fundamentals-mutations"
builder.mutationType({
  fields: (t) => ({
    updateCharacter: t.field({
      type: Character,
      args: {
        input: t.arg({
          type: builder.inputType('UpdateCharacterInput', {
            fields: (t) => ({
              characterId: t.id({ required: true }),
              biography: t.string({ required: true }),
            }),
          }),
          required: true,
        }),
      },
      resolve: (_root, { input }, ctx) => {
        if (!ctx.user) {
          throw new Error('Not signed in');
        }
        const entry = Characters.get(Number(input.characterId));
        if (!entry) {
          throw new Error(`No character with id ${input.characterId}`);
        }
        if (entry.editorId !== ctx.user.id) {
          throw new Error("Only the entry's editor can edit it");
        }
        entry.biography = input.biography;
        return entry;
      },
    }),
  }),
});
```

Four things are doing real work here. The input groups the mutation's two values into one named type. The first auth check narrows `ctx.user` from `{ id: number } | undefined` to `{ id: number }`. The lookup throws if the target doesn't exist. The permission check throws if the user isn't allowed to change *this* particular row.

## The mutation root

Like `queryType`, `mutationType` runs exactly once per schema:

```typescript
builder.mutationType({});
```

Then add fields with `mutationField` or `mutationFields` from wherever they belong:

```typescript
builder.mutationField('addCharacter', (t) =>
  t.field({
    type: Character,
    args: { input: t.arg({ type: AddCharacterInput, required: true }) },
    resolve: (_root, { input }, ctx) => addCharacter(input, ctx),
  }),
);
```

The choice between `mutationField` and `mutationFields` is the same call as for queries: per-field for module-local definitions, per-group for related batches.

## Authorization by hand

The resolver above does two different jobs in sequence. The first is the `if (!ctx.user) throw` lift that narrows `ctx.user` to a non-nullable user â€” the same move the Context page covers in [Narrowing inside a resolver](./context#narrowing-inside-a-resolver). The second is the part unique to mutations: a *row-level* permission check that decides whether this particular user may change this particular record.

The row-level check is where mutations diverge from queries. A useful refinement is to fold the existence check and the ownership check together, so the API never reveals that a record exists to a caller who isn't allowed to touch it:

```typescript
// Treat "missing" and "not yours" as the same condition, so the API
// never leaks the existence of an entry the caller can't touch.
const entry = Characters.get(Number(input.characterId));
if (!entry || entry.editorId !== ctx.user.id) {
  throw new Error('Entry not found');
}
```

Split the two into distinct throws when you want distinct messages (`Not signed in` vs `Only the entry's editor can edit it`); fold them when the existence of a record is itself privileged. Either way the shape repeats across every mutation â€” the same auth lift, the same lookup, the same permission gate â€” which is the cue to reach for a plugin.

## Returning what the client needs

Return the entity the mutation changed, so the client can re-read fields in the same round trip:

```graphql
mutation UpdateCharacter {
  updateCharacter(input: { characterId: "1", biography: "Bearer of the One Ring." }) {
    id
    biography
  }
}
```

For mutations that don't have a single natural return (a bulk delete, a batch import), wrap the result in a small payload type and return that â€” `DeleteCharactersPayload { success: Boolean!, deletedCount: Int! }` reads better than `Boolean!`.

## When to graduate to plugins

The three plugins worth knowing about for mutations:

- [`plugin-scope-auth`](../plugins/scope-auth) replaces the manual `if (!ctx.user)` and row-level checks with declarative `authScopes` on the field. Once two or three mutations share the same check, you save real LOC.
- [`plugin-errors`](../plugins/errors) turns thrown errors into typed result unions â€” your `updateCharacter` field's return type becomes `UpdateCharacterResult = Character | NotFoundError | PermissionError`, which clients can discriminate without parsing error messages.
- [`plugin-validation`](../plugins/validation) runs Zod schemas against your input before the resolver fires. Useful when the input has format constraints (`name` must be 1â€“80 chars, `email` must be a valid email).

You don't need any of them on day one. Reach for them when the manual patterns start to repeat.
