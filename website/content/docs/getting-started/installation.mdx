## Install

```package-install
npm install --save @pothos/core graphql
```

Plugins are published as separate packages (`@pothos/plugin-relay`, `@pothos/plugin-scope-auth`, …). `graphql` is a peer dependency, shared between Pothos and your server.

## TypeScript settings

Pothos is built around type safety, and turning off strict null checks breaks some of its inference. Make sure `strict` mode is enabled in your `tsconfig.json`:

```json
{
  "compilerOptions": {
    "strict": true
  }
}
```

## Hello world

A builder and a `Query` type with one field is a complete, runnable schema:

<includeregions cwd lang="typescript" meta='playground example="getting-started-first-schema-step-1"'>playground-examples/getting-started-first-schema/step-1/schema.ts#imports,builder,query,export</includeregions>

`new SchemaBuilder({})` creates the builder everything else hangs off — the options object is where plugins go later. `builder.queryType()` declares the root `Query` type ([Queries](../fundamentals/queries) covers the root types in depth), and `builder.toSchema()` produces a plain graphql-js `GraphQLSchema` that any server or tool can use.

## Adding an object type

Real schemas return objects, not just strings:

<includeregions cwd lang="typescript" meta='playground example="getting-started-first-schema-step-2"'>playground-examples/getting-started-first-schema/step-2/schema.ts#builder,character-and-query</includeregions>

`builder.objectRef<{ id: string; name: string }>('Character')` declares a GraphQL type named `Character` backed by that TypeScript shape — its backing model. `implement` defines the fields; `t.exposeID('id')` and `t.exposeString('name')` read properties straight off the backing object ([Fields](../fundamentals/fields) covers the field builder).

Because `frodo` is typed as `Character`, its resolver has to return `{ id: string; name: string }` — return the wrong shape and the file doesn't compile. Every Pothos feature builds on that same check.
