Pothos is a library for building GraphQL schemas in TypeScript. You describe your schema with a builder — objects, fields, resolvers, inputs — as ordinary values, and `builder.toSchema()` turns them into a standard graphql-js `GraphQLSchema` your server runs.

What makes that pay off is where the types come from. You start from data you already have — a database row, an API response, a domain type — and build the GraphQL object on top of it. The builder carries that type through everything built from it, so the whole schema is type-safe without you writing types twice: no SDL to sync, no codegen in the loop.

## Building from your data

Most types start with `builder.objectRef<T>()`. The generic `T` is the **backing model** — the TypeScript shape your resolvers return and Pothos hands back to you as `parent`:

```typescript playground
import SchemaBuilder from '@pothos/core';

const builder = new SchemaBuilder({});

// The data you already have
interface ICharacter {
  id: string;
  name: string;
}

const Character = builder.objectRef<ICharacter>('Character').implement({
  fields: (t) => ({
    id: t.exposeID('id'),
    name: t.exposeString('name'),
    // parent is an ICharacter, so this resolver is typed for free
    firstName: t.string({ resolve: (parent) => parent.name.split(' ')[0] }),
  }),
});

builder.queryType({
  fields: (t) => ({
    frodo: t.field({
      type: Character,
      resolve: () => ({ id: '1', name: 'Frodo Baggins' }),
    }),
  }),
});

export const schema = builder.toSchema();
```

Because `Character` is built from `ICharacter`, every resolver on it knows the shape of `parent` without a cast. Return something that isn't an `ICharacter` from `frodo` and the file won't compile. The GraphQL object and its backing model stay separate — the object is the API clients see, the model is the data you return — but the builder keeps them tied together, so the schema tracks your data as it changes.

This objectRef style is what the rest of the guide uses. Pothos also supports class-backed types and types registered by name on the builder's generics — [Object types](../fundamentals/objects) covers all three.

## How it compares to schema-first

Schema-first tools — GraphQL Tools, Mercurius, raw `graphql-js` — start from an SDL string and ask you to write resolvers that match it, with codegen bridging the SDL back to TypeScript types. Pothos runs the other direction: your TypeScript is the source of truth, and the SDL is a derived artifact of `toSchema()`. There's one description of the schema, so the types can't drift from it.

## Where plugins fit

Core gives you objects, fields, inputs, interfaces, unions, enums, and scalars — enough for a complete schema, with `graphql` as its only peer dependency. Everything else is a plugin that adds methods to the same builder:

- **[Relay](../plugins/relay)** — connections, nodes, and global IDs following the Relay spec.
- **[Scope auth](../plugins/scope-auth)** — declarative authorization checks on fields and types.
- **[Errors](../plugins/errors)** — model expected failures as part of the schema instead of throwing.
- **[Validation](../plugins/validation)** — validate arguments and inputs with Zod or Valibot.
- **[Dataloader](../plugins/dataloader)** — batch and cache loads to avoid N+1 queries.
- **[Prisma](../plugins/prisma) / [Drizzle](../plugins/drizzle)** — define objects straight from your ORM models.
- **[Federation](../plugins/federation)** — build subgraphs for a federated gateway.

A plugin isn't a separate framework bolted on. It's more methods on `t` and the builder, typed the same way, so `t.prismaObject` and `t.connection` sit next to `t.field` and share the same inference. You add plugins as the schema needs them, and they compose with each other.
